### 2.2 Diagnostic critique du modèle SIEM Windows
En regardant `siem_windows/results/metrics.json` j'ai trouvé un **problème plus grave** que ce qui était documenté :

| Indicateur | Valeur | Interprétation |
|---|---|---|
| ROC-AUC test | **0.46** | Pire que l'aléatoire (modèle anti-corrélé) |
| F1 cross-val (train) | 0.89 | Bon sur train… |
| F1 test | 0.76 | …mais piégé : c'est juste 88/144=61 % de positifs (recall=1.0, precision=0.61, le modèle prédit TOUT comme attaque) |
| Recall normaux test | 0.0 | 0 vrai négatif sur 56 normaux |
| Top feature | `total_events` (0.17) | Pas de leakage, mais 11 features mortes (importance=0) |

**Diagnostic profond :**
1. **Distribution shift day1 → day2** : day2 est 3× plus bruyant que day1 (médiane 4479 vs 3171 events / fenêtre normale).
2. **Inversion de signe sur day2** : les fenêtres normales day2 ont MORE events (mean=22 632) que les fenêtres attaque (mean=16 142) — le modèle a appris la mauvaise direction.
3. **Trop peu de données** : 132 train + 36 features = overfitting garanti.
4. **11/36 features mortes** : EventIDs rares qui polluent.

### 2.3 Stratégie validée et appliquée

**Stratégie « GroupShuffleSplit par host + features pruned + class_weight »** (option choisie par toi) :
- Concaténer day1+day2 (276 fenêtres) puis split par host (3 hosts train, 1 host test = NEWYORK).
- Drop features avec importance < 0.005 (15 features supprimées).
- RF régularisé : `max_depth=5, min_samples_leaf=3, class_weight='balanced'`.
- Calibration isotonic + seuil F1-optimal.

### 2.4 Modifications de code
- `siem_windows/preprocessing/preprocess_siem.py` :
  - Import `GroupShuffleSplit`
  - Merge day1+day2, split par host (test=25 %)
  - Manifest v2 avec `train_hosts` / `test_hosts`
- `siem_windows/training/train_siem.py` :
  - Pruning auto features mortes (importance < 0.005)
  - `max_depth=5, min_samples_leaf=3`
  - Logs ASCII (suppression `±` qui pétait sur cp1252)
- `lateral_movement/preprocessing/preprocess_lateral.py` :
  - Fix encodage Unicode (`→` → `->`)
  - Split stratifié par label (chaque label split séparément en GroupShuffleSplit) pour garantir des deux classes en train ET test (avant : test=100 % positifs).
